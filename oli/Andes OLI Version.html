Building the Andes OLI Course


This document describes how to build the OLI Andes-based physics course content from the Andes source code tree. Formally, what you are building is called a content package. The content package includes all resources needed for the content. This content package is deployed onto an OLI server. Once deployed, instructors or developers with the appropriate access may then create course sections which are instances of this content package.

Accessing OLI

You must obtain a CMU andrew account. The OLI folks must give your account permission to access the OLI servers.

OLI course development involves the following servers:
  • The CVS repository server: oli-cvs.andrew.cmu.edu
  • The development server (ODIN):  odin.web.cmu.edu
  • The QA server: oli-qa.andrew.cmu
  • The production server: www.cmu.edu/oli/
The front page of ODIN links to some documentation on OLI development processes. A course author can freely build, deploy and update a private version of the content package on the development server and then build test courses using that package. Moving the course to QA or production is done by OLI staff at the appropriate time.

OLI requires all source for building a course to be stored in its own CVS repository.  The OLI-CVS host name is actually the same as odin, the development server. The bulk of the source to go into this repository are OLI problem pages and related problem files which are generated automatically from the Andes source tree. Other portions like the course introduction page must be created or modifiedby hand.

Checking out from OLI-CVS

The CVS module containing the OLI physics course source is named "intro-physics-1.0".  To check out the main branch use a command like the following:

                cvs -d :ext:andersw@oli-cvs.andrew.cmu.edu:/oli/cvs/content/lrdc/intro_physics co intro_physics-1.0

replacing the "andersw" with the appropriate user id. This will create a directory named intro_physics-1.0 containing the OLI source tree. I usually do this within the "oli" subdirectory of the Andes working directory.

Note: the "1.0" at  the end of the CVS project name is meaningless, and is potentially misleading, since the 1.0 here never changes as new editions of the course are developed. There is a different version number that identifies the OLI content package version. The real content package version number is set inside the sources (see below).  This content package version number functions within OLI somewhat like an edition number on a textbook: courses built on one version of the content package get patched as that version of the content package is updated, but such courses are never upgraded to use a different version of the content. The content package version has usually been incremented each semester though this is not strictly necessary.  It now stands at 1.8.

OLI Source structure

The OLI source has two main subdirectories:
    content - holds the course content
    organizations - holds  the  files specifying possible organizations of the course

It also contains a "tools" directory  which  we do  not modify.

The content tree

package.xml
The root of the content tree contains the file package.xml which describes the content package. The most important thing in this is the version identifier. To create a new version the version number must be changed by hand.
standard subdirectories
Within the content tree, files are grouped by type into subdirectories named for the MIME type of files they contain.
    x-oli-workbook_page  -- OLI workbook pages
    x-pitt-lrdc-andes1-assignment -- xml problem descriptor files (see below)
    webcontent -- any content files that can be served without special handling by a web server: gifs, jpegs, wmvs, whatever.

For andes-specific files, we use a subdirectory within a webcontent folder:
    webcontent/andes -- andes-problem files. This includes the prb files for the problems as well as gifs or jpegs used.
unit and module structure
The content directory is structured into a subdirectory for one unit which contains two module subdirectories within it which parallel the original unit and module structure of the course. These are named
  content
    _u1_intro_physics
          _m01_mechanics
          _m02_em

The key point is that the mechanics-related files have to go into the _m01_mechanics tree while the E&M files have to go within the _m02_em tree. Both are contained in one big "unit" which is essentially the whole course.

Note: The hierarchical structure with unit and module directories was once required by the OLI system but is not required any longer. So this sturcture could in principle be simplified so as to include all problem files in one directory. But it is a nuisance to change structure once the CVS is set up. The existing script assumes this directory structure.

The _u1_intro_physics unit directory contains the course introduction page, and also an instructor-only information page. One might expect supporting webcontent for these pages to be contained within the _u1_intro_physics directory. However, for obscure historical reasons, supporting webcontent for these pages are contained ina webcontent directory at the root of the course (within the content directory), rather than within the _u1... subdirectory.

The organizations tree

This contains one subdirectory for each organization that an instructor may select when building a course from this content package. An organization file defines a table of contents structure over the course content, and may also customize the stakes or gating requirements. We provide three:
    default -- the default, full course
    em -- the E&M only course
    mechanics -- the mechanics only problems.

The organizations are defined in an xml file using OLI's simple format, in which it is only necessary to list the relevant modules (not each individual problem).  The simple organization files change very rarely so are not automatically generated. The only change that could be necessary would be to add a new problem set, in which case the necessary changes should be obvious using the existing

Organization files do contain a content package version number which must match the version number in the content package. So, a change to the version number must also be applied to each organization.xml file.

The gating requirements defined for Sandy Katz's experiment require using the advanced format organization files. This involves listing each problem with its prerequisites. Assuming this will not be required it is not described here.
