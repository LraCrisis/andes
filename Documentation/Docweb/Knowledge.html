Knowledge

(as of 4/5/01)

Authors: Collin Lynch

The Knowledge defines datastructures such as Problem, Entry Proposition, etc that are loaded and used by the Solution Graph Generator and the Help system.
Ultimately this file will link to those files but at present only a few files are connected.

Files:

  • AMFile.cl -- Andes Module file for the Knowledge database.
  • Bubblegraph.cl -- Defines the Bubblegraph, Qnode Enode and other structures.
  • Nogood.cl -- Nogood definitions.
  • Ontology.cl -- Definition of expression objects.
  • OpApp -- Operator application structure.
  • Operator.cl -- Definition of an operator struct.
  • Problem.cl -- Definition of a problem struct.

Small data structures:

  • Entry proposition -- This is a proposition in the KB that happens to represent a entry.  For instance,

    • (vector puck (at (force puck earth weight) (during 1 2)) 270)
    represents a weight vector.  The KB operators create these, and they get stored in the PSM nodes.  Then they get extracted and put into System entries (see below).  The Student entries have them as well, but they are created by domain-specific code in the Entry interpreter that knows how to translate information from the Workbench into entry propositions with the correct syntax.  The list of current entry propositions is here.
  • Student entry -- represents a workbench action where the student made a problem solving entry, such as writing an equation or drawing a vector.  Fields are
    • Identifier -- atom assigned by the workbench.  When the workbench says something is deleted, this is the ID that it uses.
    • Proposition -- an entry proposition built by the Entry interpreter.
    • Correctness -- whether the entry is correct, inefficient, dead-path, forbidden or incorrect
    • Interpretation -- If the entry is correct, this is a set of system entries, this is the set of the system entries that correspond to it.  Its a set because a compound student equation corresponds to a set of primitive equations.  If the entry is incorrect, this is NIL or an error interpretation produced by an error handler.
    • Correct-interpretations -- If the entry is correct, all logically possible interpretations are stored here.  One of these is selected by the cognitive load procedure and put into the Interpretation slot by the Entry Interpreter.
    • Error-interpretations -- If the entry is incorrect, all interpretations produced by the error handlers are stored here.  One of these is selected by What's wrong help and put in the the Interpretation slot.
    • Verbatim -- If the entry is an equation, this records the student's string exactly as they typed it before it was parsed.  This is useful for some error handlers, such as Equation-used-as-method-name.
  • System entry -- represents a entry made by the problem solver.  The same entry may be made by different operator applications.  However, we assume that when this occurs, the operator applications are equalp and their prerequisites are equalp.  Thus, the help generated from each operator application is the same, so we don't need to disambiguate them.
    • Index -- the entry's position in the Entry array.  This number is used for communication with the algebra module.
    • Proposition -- an entry proposition that is contained in all the PSM nodes.
    • Sources -- a set of Cognitive steps that represent the operator applications that produced this entry.
    • Op-app -- an operator application, which is the atomic name of the operator cons'ed to the values of all its variables.  The order is determined by the order of variables in the Vars field of the operator (see below).  The value of this field should equal the values of all the op-app fields of the sources.  It is repeated here just in case a certain guarantee fails (i.e., that equal entry propositions imply equal operator applications; see discussion) and we have to work around it.
    • Correctness -- one of correct, inefficient, incorrect, dead-path or forbidden.  This is how to treat the system entry as a whole.  It may differ from the Correctness fields of the sources, as described here.
    • Prereqs -- a set of system entries.  This possibly-empty set is the system entries that must be entered before the student is allowed to enter this system entry.  An operator application X requires an op app Y if they are in the same PSM, and X precedes Y in the Path representation of the partially order of steps.  However, the pre-reqs of a system entry are other system entries, not nodes in the PSM graph, because we assume that if two operator applications produce the same entry proposition, then their pre-requisites are also equivalent (i.e., produce the same entry propositions, and thus correspond to the same system entry).  See discussion.
  • Tutor turn  -- represents the tutor's turn in the dialog.  Returned by many modules to the Manager, which then forwards on the appropriate info to the Workbench, while remembering the tutor turn so that it can interpret the next student turn in the dialog.  Fields are:
    • type -- one of dialog, play-minilesson, color-red, color-green, delete, end-dialog
    • text -- a string.  If this is a dialog or delete type, then will play this string.  If it is a minilesson, then this is the file name.
    • menu -- the name of a workbench menu or edit control
    • responder -- a one-argument function that is applied to the student's turn in order to respond to it.
  • Operator -- represents a primitive piece of physics knowledge.  Used extensively in the KB but referenced by other modules as well.  Fields are
    • name -- an atom
    • vars -- a list of all the variables used in the operator [Collin: I think you need to add this. -- KVL, 4/3/01]
    • args -- list of ? variables.  These are used inside the solver to prevent repetition of operator applications.  Args is a subset of Vars, as discussed here.
    • preconditions -- list of propositions and executable preconditions
    • effects -- list of propositions
    • hints -- list of hints
    • features -- possibly empty list that could include Unordered,...
    • cognitive load -- an integer.  Can be missing, in which case the cognitive load defaults to 2.
  • Hint -- represents a hint to be given to the student.  Written by authors and included as part of an operator.  Passed around by the what's next and other modules that plan how to give hints. Details  here.
