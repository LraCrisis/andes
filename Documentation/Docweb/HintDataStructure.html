The Hint data structure

(as of 3/16/01)

In Andes 2, physics knowledge is represented by operators.  Each operator has a Hints slot, which contains an ordered list of hints, where a hint is an instance of the following structure

(defstruct hint
    format
    functions
    string
    args)

The format of the  hint is either STRING, MINILESSON or KCD.  The functions field contains a set of atoms:
 

  • teach -- this hint attempts to teach the relevant operator or concept. For instance, a string hint that teaches the normal force direction would be "The direction of the normal force is always perpendicular to the surface that exerts it.  This is because ...."
  • point -- a "pointing" hint in the style of those identified by Hume et al.  It gives some information away in the hopes that this will trigger the student to recall or reconstruct the rest.  For instance, "Think about the direction of the normal force."
  • goal -- Prompts the students by mentioning the goal of this operator., e.g., "You need to draw a force due to the surface."
  • bottom-out -- Tells the student exactly what to do.
We can add more functions later.  For instance, Recap might be a hint that is presented after a problem is over, and Remind could be a teaching hint that is presented only if the student has already seen a teaching hint on this topic.  The  functions field is a *set* of functions because the same hint might be useful for multiple purposes.  For instance, a Remind hint and a Recap hint might be the same.

The string field is the string to be printed if this is a STRING hint. If the hint is a MINILESSON or KCD hint, then the string field contains an string that names the file containing the minilesson--this file is loaded by the workbench inorder to play the minilesson.  If the hint is a KCD hint, then the string is names the KCD.  This could be a file name or an index into some in-core list of KCDs.

The args field represents the arguments of the hint.  The form of the arguments depends on whether the hint is in the Hints slot of an operator, or whether it is being passed around in the dialog system.  When it is in the Hints slot of an operator, the args slot contains set of FOL terms that include variables.  For instance, the following is a string hint as it appears in the Hints field of an operator:

(string (point) "Think about the forces acting on ~a ~a." (?body ?time))

The terms ?body and ?time are not bound yet, but will be bound when the operator is applied.  When dialog system wants to find out what hints are available for a particular path-node in the PSM graph, it calls a function that not only returns the hints, but replaces the variables in the args field with their bindings.  Thus, the same hint when the dialog system first gets ahold of it would be

(string (point) "Think about the forces acting on ~a ~a." (Car1 (during 1 2)))

When the hint is eventually generated, these terms are converted into text, and the text replaces the ~a codes in the string.  Thus, the eventual hint shown by the workbench would be "Think about the forces acting on the car during time 1 to time 2."

Currently, KCDs  do not accept arguments.  Minilessons may, but I'm not sure because I don't know how well the old template-based minilessons that Anders implemented would fit in our new system.  Someone needs to resurrect that code.
